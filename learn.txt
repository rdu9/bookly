migrations folder:
version/ - every file here is one recorded database change
env.py - tells alembic how to connect to the database, reads DATABASE_URL from config.py instead of taking it from alembic.ini, which would have hardcoded it.
README - generated by alembic, just some auto-generated information
script.py.mako - alembic template for generating new migration files, never touch manually

auth folder:
__init__.py - makes the auth folder an importable module
models.py - here the User table for the db is created, same as the books db but with the exclude=True on password_has for security reasons, nothing really changed, just a new table
routers.py - here is where the HTTP requests are handled , on the specific api. then calls to service.py with the users data and returns whats needed
schemas.py - this is how the payload should be sent, its the same of the books schemas, but with the exclude=True on password_has for security reasons
service.py - handles all database logic, just as the books services.py, but this time it acceses the User table of the database, every call is made from routers.py as a shortcut
utils.py - this is where the password hashing happens, checking , the actual hashing process, everything, very important for security, and it created and verifies tokens, which are essential for the JWT login scheme
dependencies.py - this is where the JWT token gets checked, so no one can acces a private api without the proper authorization

books folder:
__init__.py - makes the books folder an importable module
book_data.py - useless, now im using DB there its just a normal dictionary with hardcoded books
models.py - here the DB tables get created 
routes.py - here all the HTTP requests are handled, reading the request, validating the payload via schemas, and returning the response, while service.py handles the db layer
services.py - handles all database logic, routes call functions from this file to read, create, update or delete data in the db
schemas.py - this is how the payload should be sent, i think of this like a check , a verify system, as i said earlier. this is used in services.py as type hints, no check , the check happens in routes.py

db folder:
__init__.py - this folder just makes db an importable module
main.py - this is where the bridge between postgres and python lives, and it defines the get_session, which allows every route to acces the database

root folder:
__init__.py - this is the heart of everything, where the main app lives and where the db is initialized,  withing the lifespan.
config.py - this is where the database_url lies, this is put into the Settings class to be easily accesible, and removing the need for os.getenv("database_url")
.env - the private file with the private info
.gitignore - what is hidden from github
old_main.py - to be ignored, my old main.py file
requirements.txt - all the packages needed to run the 
alembic.ini - this is the alembic config file, tells the alembic where migration folder is and how to format logs. everything else doesnt really matter
